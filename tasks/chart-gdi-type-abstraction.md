# Chart Engine: GDI+ Type Abstraction ŌĆö Progress

**Parent plan:** [`chart-cross-platform-implementation.md`](chart-cross-platform-implementation.md)
**Design, namespaces, and recurring patterns:** [`docs/rendering-abstractions.md`](../docs/rendering-abstractions.md)
**Decisions:** [`docs/decisions.md`](../docs/decisions.md)

**Goal:** Replace concrete GDI+ resource types (`Pen`, `Brush`, `Font`, `GraphicsPath`, ŌĆ”) with backend-agnostic interfaces + an abstract factory, so the chart engine can run on GDI+ (Windows) and SkiaSharp (Linux/macOS).

## Open items (detail needed to resume)

**B1b / shared `pen`/`solidBrush` fields on `ChartGraphics`.** Deliberately not retyped ŌĆö superseded by giving new conversions their own local `IPen`/`IBrush` via `resourceFactory` instead (the pattern used throughout B2). `pen` feeds `DrawPath(pen, path)` (`ChartGraphics.cs:2295`, no mixed `DrawPath(IPen, GraphicsPath)` overload by design); `solidBrush` is interchanged with `GetGradientBrush`/`GetHatchBrush`/`GetTextureBrush` results in one `Brush brush` local (`ChartGraphics.cs:2052-2063`, `2202-2224`). If ever revisited: `DrawPathAbs` (`ChartGraphics.cs:2175`) has 6 real callers (`PolylineAnnotation.cs`, `CalloutAnnotation.cs`, `ArrowAnnotation.cs`, `FunnelChart.cs`, `Borders3D/SunkenBorder.cs`, `Borders3D/EmbossBorder.cs`) that would ripple.

**B2 holdouts.** `FunnelChart.cs`'s `DrawFunnelCircularSegment` (3D branch) builds a `graphicsPath` via a concrete-only `AddEllipseSegment` helper consumed directly by real `System.Drawing.Graphics.FillPath/DrawPath`, not `IChartRenderingEngine` ŌĆö stays concrete. `ChartGraphics3D.cs`'s `Draw3DSurface`/`Draw3DPolygon` had their `frontLinePen` field entanglement resolved via a narrow `BridgeFrontLinePen` helper, but the rest of both method bodies remain concrete by design (converting further wasn't judged worth it once the one real consumer was bridged). Map engine (`Microsoft.Reporting.Map.WebForms`) and Gauge engine each have their own separate, parallel GDI+ pipeline ŌĆö out of scope here; Gauge tracked in `gauge-gdi-type-abstraction.md`, Map's scope is an open decision.

**D2 ŌĆö backend selection.** The naive fix (swap `ChartRenderingEngine.RenderingObject`'s backend by `RuntimeInformation.IsOSPlatform`) would be reachable but pointless: the real production entry point, `ChartImage.GetImage(float) : Bitmap`, unconditionally downcasts to `GdiRenderSurface` and returns a hard GDI+-typed `Bitmap` *before* `RenderingObject` is ever consulted ŌĆö throws on Linux regardless of backend selection, and `GetImage`'s own return type is a public API contract a Skia backend still couldn't satisfy (`Bitmap` construction itself fails on Linux per the Phase 0 spike). One independent, safe fix landed alongside this investigation: `Chart.Save(Stream, ChartImageFormat)` now goes through `ChartImage.SaveImage(Stream, ChartImageFormat, float)`, which calls the previously-unused `IRenderSurface.Encode(Stream, ChartImageFormat)` instead of a raw `Bitmap.Save`. **Real blocker for D2/E1:** `Paint(Graphics)`/`ChartGraphics.Graphics` must stop being GDI+-typed before backend selection has anything real to select between.

*Scoping pass (2026-07-22) found the blocker's real call-site count much smaller than the "~70 painter files" framing implied ŌĆö B2 already did the heavy lifting. Confirmed ~49 total sites across 14 files reading `.Graphics` off `ChartGraphics`/`IChartRenderingEngine` and calling raw `System.Drawing.Graphics` members directly. Converted so far (~41 sites):*
- *`ImageLoader.GetAdjustedImageSize(Image, Graphics, ref SizeF)`'s `Graphics` parameter was already dead (unused since the 96-DPI-baseline decision) ŌĆö dropped from both overloads' signatures; fixed all 28 call sites across `Axis.cs`, `ChartGraphics.cs`, `ImageAnnotation.cs`, `LegendCell.cs`, and `ChartTypes/{BarChart,BoxPlotChart,ErrorBarChart,PointChart,RadarChart,StockChart}.cs`.*
- *Added `float GetDpiX()` and `CompositingQuality`/`InterpolationMode` properties to `IChartRenderingEngine` (same pattern as the existing `SmoothingMode` property ŌĆö GDI+-enum-typed by design, backed by the live `Graphics` on Gdi/Svg, `NotReachable` stub on the Skia spike). Converted the 6 remaining `.Graphics.DpiX` reads (`ChartGraphics.cs`, `ChartPicture.cs`, `LegendCell.cs`├Ś3, `LegendItem.cs`) and `LegendCell.cs`'s 6-site `CompositingQuality`/`InterpolationMode` save-restore block.*

*`Paint` chain conversion (2026-07-22, same session): added `IChartRenderingEngine.BindSurface(IRenderSurface)` ŌĆö each backend downcasts internally to its own concrete surface type (`GdiGraphics`/`SvgChartGraphics` both do `((GdiRenderSurface)surface).NativeGraphics`; `SkiaChartGraphics` stubs `NotReachable`, consistent with its other unimplemented members), so the downcast moved from 4 call sites into the adapters that are allowed to know their own concrete types. `ChartPicture.Paint`'s 2-arg and 8-arg overloads got `IRenderSurface`-typed siblings (dual-overload, per this doc's own convention) that call `chartGraph.BindSurface(...)` instead of `chartGraph.Graphics = ...`; the original `Graphics`-typed overloads are kept permanently (not deprecated) since `ChartImage.SaveIntoMetafile` genuinely needs to hand in a Metafile-backed `Graphics` with no `IRenderSurface` equivalent. Both overload pairs delegate into a shared private `PaintCore(..., Action bindGraph)` to avoid duplicating the ~300-line body ŌĆö `bindGraph()` must run at the exact point it did before (after the SVG `Open(...)` call, which swaps in a fresh `SvgChartGraphics`, and before anything else touches the surface). Converted all 4 real callers: `ChartImage.GetImage`, `ChartImage.SaveImage`, `ChartImage.GetSvgImage`, `ChartPicture.Select` now pass their `IRenderSurface` straight through instead of downcasting to pull out `.NativeGraphics`. Also fixed one bonus site found in `Paint`'s own body (`chartGraph.Graphics.SmoothingMode` ŌåÆ `chartGraph.SmoothingMode`).*

*Tail closed out (2026-07-22, same session):*
- **`TextAnnotation.GetContentPosition`**'s standalone `new Bitmap(...)` + `Graphics.FromImage(...)` construction (for measuring annotation content size before any chart-attached `ChartGraphics` exists) now routes through `chart.chartPicture.renderSurfaceFactory.CreateRasterSurface(...)` + `chartGraphics.BindSurface(...)`, same pattern as `ChartImage.GetSvgImage`. `IRenderSurface.Dispose()` replaces the old separate `graphics.Dispose()`/`image.Dispose()` calls.
- **`SelectionManager.Annotation.DrawSelection(Graphics g)`** (`Utilities/SelectionManager.cs`) ŌĆö the `graph.Graphics = g` reassignment was a provable no-op (its one caller, `DrawSelection()`, always passes `graph.Graphics` right back to itself); removed, with a comment on why `g` is still accepted (kept for the caller's shape, not to compatibility-preserve behavior that no longer exists).

*Remaining (permanent, by design ŌĆö not further work):*
- **`FunnelChart.DrawFunnelCircularSegment`** (3D branch) and **`ChartImage.SaveIntoMetafile`** (EMF/HDC construction, now the sole caller of the retained `Graphics`-typed `Paint` overloads) ŌĆö the two already-documented permanent holdouts (B2 holdouts / D1 notes).
- `ChartImage.GetImage`/`SaveImage`'s background-rectangle fill still does its own `GdiRenderSurface` downcast for a raw pre-paint `graphics.DrawRectangle` call ŌĆö not part of the `Paint` chain (it draws before `Paint` runs), could become an `IRenderSurface.Clear`/`FillBackground` method in a future increment but wasn't judged worth it for this pass.

**D3 ŌĆö remove temporary concrete overloads.** Blocked on: the whole `ChartGraphics3D.cs` 3D subsystem (includes `RangeChart.Draw3DSplinePolygon`'s concrete `GraphicsPath`/`Pen`/`Brush` usage ŌĆö checked 2026-07-22, genuinely part of this holdout, not separable), `MapAreasCollection.cs` (scope undecided), and the entire untouched Gauge engine.

*Correction (2026-07-22):* the two items previously listed here ŌĆö "`RangeChart.cs`'s shadow-region block" and "`CalloutAnnotation.cs`'s cached geometry helpers" ŌĆö turned out to already be resolved on inspection, not open blockers. `RangeChart.DrawLine`'s actual shadow-offset code (`ShadowColor`/`ShadowOffset`) is fully `IClipRegion`/`IBrush`/`IPen`-typed already. `CalloutAnnotation.GetCloudPath`/`GetCloudOutlinePath` are self-contained legacy geometry generators (a hand-authored Bezier cloud shape, static/cached) whose only call sites already bridge them at the sink via `PathPoints`/`PathTypes` into `IGraphicsPath` ŌĆö exactly the bridge-at-sink pattern this doc's conventions section describes, already applied. Removed both as stale.

**E1 ŌĆö Skia backend real wiring.** The spike's own comment (`SkiaChartGraphics.cs`, "everything below is intentionally unreachable") says the GDI+-typed half of `IChartRenderingEngine` is only reachable because `ChartGraphics.cs` and painter files still allocate concrete `Pen`/`Brush`/`Font`/`GraphicsPath` locals and feed them into the GDI+-typed overloads. Scoping pass (2026-07-22) confirmed this is still true and sized it: **independent of and much smaller than D3** ŌĆö D3 targets removing concrete overloads *everywhere* including the 3D subsystem, Map, and the entire Gauge engine; E1 only needs the reachable 2D chart-paint path clean, a bounded ~100-150-site cleanup concentrated in `ChartGraphics.cs`'s shared helpers (high leverage, since ~70 painter files draw through them) plus a dozen `ChartTypes/*.cs` files. `BarChart.cs`/`ColumnChart.cs`/`BoxPlotChart.cs`/`StockChart.cs` are already fully clean (zero concrete allocations feeding GDI+-typed calls). Concrete example that previously blocked even a simple bar/column chart: `ChartGraphics.FillRectangleShadowAbs` (whenever `shadowColor`/`shadowOffset` are set) and `DrawMarkerAbs` (whenever point markers are enabled) ŌĆö both now converted (this session).

Converted so far:
- **`ChartGraphics.FillRectangleShadowAbs`** (both the soft-shadow `PathGradientBrush`/`GraphicsPath` branch and the hard-shadow `SolidBrush` branch) ŌĆö retyped in place (no signature change: params are `Color`/`float`/`bool`, not `Pen`/`Brush`) to use `resourceFactory.CreateSolidBrush`/`CreatePath`/`CreatePathGradientBrush` and the interface-typed `FillRectangle(IBrush,...)`/`FillPath(IBrush,IGraphicsPath)`/`DrawCircleAbs(IPen,IBrush,...)` (the last already had an interface-typed sibling from a prior conversion).
- **`ChartGraphics.DrawMarkerAbs`**'s shape-marker branch (Star/Circle/Square/Cross/Diamond/Triangle ŌĆö everything except the `markerImage.Length > 0` image-marker branch, deliberately left concrete since `ImageLoader`'s pipeline is GDI+-only and unreachable on Skia regardless, per the doc's Namespaces section) ŌĆö same in-place retype, same resource-factory pattern.
- **`ChartGraphics.DrawPieRel`** (the `PieChart`/`SunburstChart` pie/doughnut chokepoint, ~70 painter files indirectly benefit since it's the shared pie-fill path) plus its two private helpers **`DrawPieGradientEffects`** (Concave/SoftEdge pie surface effects) and **`DrawPieSoftShadow`** ŌĆö all retyped in place using the already-existing `*Resource`-suffixed Milestone-B2 siblings (`GetHatchBrushResource`/`GetTextureBrushResource`/`GetGradientBrushResource`/`GetPieGradientBrushResource`) plus `resourceFactory.CreatePath`/`CreatePen`/`CreatePathGradientBrush`/`CreateLinearGradientBrush`. Required adding `Blend`/`InterpolationColors` to the shared `IPathGradientBrush` interface (previously only on `ILinearGradientBrush`, an omission from the original Appendix A.5 count) and implementing them on both the Chart and Gauge engines' `GdiPathGradientBrush` (the two engines share the same `Microsoft.Reporting.Rendering.IPathGradientBrush`/`IBrush.cs`). `SunburstChart.RenderSlice` is the one real caller of the `out GraphicsPath` overload (it threads the path into `CanLabelFitInSlice`/hit-testing, out of scope for this increment) ŌĆö kept as a concrete-typed dual-overload sibling that bridges at the sink via `GdiGraphicsPath.NativePath`, Gdi-only for now (a documented, scoped residual, not a blocker for the mainline 2D-chart-paint path).

Remaining (not yet converted ŌĆö the bulk of the ~100-150 estimate):
