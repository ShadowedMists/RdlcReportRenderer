# Chart Engine: GDI+ Type Abstraction — Progress

**Parent plan:** [`chart-cross-platform-implementation.md`](chart-cross-platform-implementation.md)
**Design, namespaces, and recurring patterns:** [`docs/rendering-abstractions.md`](../docs/rendering-abstractions.md)
**Decisions:** [`docs/decisions.md`](../docs/decisions.md)
**Known gaps:** [`docs/platform-support.md`](../docs/platform-support.md)

**Goal:** Replace concrete GDI+ resource types (`Pen`, `Brush`, `Font`, `GraphicsPath`, …) with backend-agnostic interfaces + an abstract factory, so the chart engine can run on GDI+ (Windows) and SkiaSharp (Linux/macOS).

## Open items (detail needed to resume)

**B1b / shared `pen`/`solidBrush` fields on `ChartGraphics` — done (2026-07-23).** Previously deferred as "superseded" since new conversions all got their own local `IPen`/`IBrush` via `resourceFactory` instead of touching the shared fields. Revisited and completed:
- `pen` retyped `Pen` → `IPen`. Its 2 remaining direct consumers (`DrawLineAbs`, the concrete `FillRectangleRel`) needed no body changes — C# overload resolution picked the existing `DrawLine(IPen,...)`/`DrawRectangle(IPen,...)`/`DrawCircleAbs(IPen,IBrush,...)` interface overloads automatically once the field's static type changed.
- `FillRectangleAbs` (concrete) and the 13-arg `DrawPathAbs(GraphicsPath,...)` both used `pen`/`solidBrush` together with concrete-only image/brush-interchange logic that couldn't just retype in place. Both already had interface-typed siblings with identical (`FillRectangleAbs`/`FillRectangleAbsResource`) or bridgeable (`DrawPathAbs(GraphicsPath,...)`/`DrawPathAbs(IGraphicsPath,...)`) signatures, so both became one-line forwards instead of duplicating logic: `FillRectangleAbs` now calls `FillRectangleAbsResource` directly (identical param list); `DrawPathAbs(GraphicsPath,...)` bridges via `resourceFactory.CreatePath(path.PathPoints, path.PathTypes)` (the same bridge used elsewhere, e.g. `Axis.cs`) and calls `DrawPathAbs(IGraphicsPath,...)` — avoiding the ripple into its 6 real callers (`PolylineAnnotation.cs`, `CalloutAnnotation.cs`, `ArrowAnnotation.cs`, `FunnelChart.cs`, `Borders3D/SunkenBorder.cs`, `Borders3D/EmbossBorder.cs`) that a signature change would have caused.
- `solidBrush` ended up with no remaining readers once both consumers above were converted — removed entirely (field, ctor init, `Dispose` call) rather than kept as dead weight.
- One real gap surfaced during the retype: 3 external files (`LineAnnotation.cs`, `PolylineAnnotation.cs`, `CalloutAnnotation.cs`) read `graphics.pen` directly to set `CustomStartCap`/`CustomEndCap` for arrow-anchored lines — the documented C3 `AdjustableArrowCap` permanent holdout, previously relying on the field being concrete `Pen`. Rather than leave these 3 callers broken, added `CustomLineCap CustomStartCap`/`CustomEndCap` to the shared `IPen` interface itself (same precedent as already exposing other GDI+-only types through `IPen`/`IPathGradientBrush`, e.g. `DashStyle`/`LineCap`/`Blend`): real pass-through on both `GdiPen` implementations (Chart's and Gauge's — `IPen` is shared), `NotSupportedException` on `SkiaPen` (matching the existing unsupported-gradient/hatch-pen-brush precedent). This is additive, not a reversal of the C3 holdout — `AdjustableArrowCap` itself still has no cross-platform equivalent, only its accessor moved onto the interface.
- Verified: `dotnet build --no-incremental` 0 `error CS` lines, `VisualRegressionTests` 80/80 (zero pixel diffs), `Chart.Rdl.Tests` 1/1.

**B2 holdouts.** `FunnelChart.cs`'s `DrawFunnelCircularSegment` (3D branch) builds a `graphicsPath` via a concrete-only `AddEllipseSegment` helper consumed directly by real `System.Drawing.Graphics.FillPath/DrawPath`, not `IChartRenderingEngine` — stays concrete. `ChartGraphics3D.cs`'s `Draw3DSurface`/`Draw3DPolygon` had their `frontLinePen` field entanglement resolved via a narrow `BridgeFrontLinePen` helper, but the rest of both method bodies remain concrete by design (converting further wasn't judged worth it once the one real consumer was bridged). Map engine (`Microsoft.Reporting.Map.WebForms`) and Gauge engine each have their own separate, parallel GDI+ pipeline — out of scope here; Gauge tracked in `gauge-gdi-type-abstraction.md`, Map's scope is an open decision.

**D2 — backend selection.** The naive fix (swap `ChartRenderingEngine.RenderingObject`'s backend by `RuntimeInformation.IsOSPlatform`) would be reachable but pointless: the real production entry point, `ChartImage.GetImage(float) : Bitmap`, unconditionally downcasts to `GdiRenderSurface` and returns a hard GDI+-typed `Bitmap` *before* `RenderingObject` is ever consulted — throws on Linux regardless of backend selection, and `GetImage`'s own return type is a public API contract a Skia backend still couldn't satisfy (`Bitmap` construction itself fails on Linux per the Phase 0 spike). One independent, safe fix landed alongside this investigation: `Chart.Save(Stream, ChartImageFormat)` now goes through `ChartImage.SaveImage(Stream, ChartImageFormat, float)`, which calls the previously-unused `IRenderSurface.Encode(Stream, ChartImageFormat)` instead of a raw `Bitmap.Save`. **Real blocker for D2/E1:** `Paint(Graphics)`/`ChartGraphics.Graphics` must stop being GDI+-typed before backend selection has anything real to select between.
