Sibling effort to [`chart-gdi-type-abstraction.md`](chart-gdi-type-abstraction.md), applying the same Ports & Adapters pattern to `Microsoft.Reporting.Gauge.WebForms` (~30,000 lines / 253 files, never previously touched by the Chart migration).

Scoping found the Gauge engine already has a pre-Milestone-A-equivalent seam (`IGaugeRenderingEngine`/`RenderingEngine`/`GdiGraphics`) and a smaller, more concentrated GDI+ surface than Chart's (~249 construction sites/~30 files vs. Chart's ~1000/~24) â€” critically, zero `GraphicsPath.Widen`/`GraphicsPathIterator`/custom-arrow-cap sites, so several of Chart's trickiest gaps don't recur here.

## Open items (detail needed to resume)

1. ~~**`HotRegionList.SetHotRegion`/`AddHotRegion`**~~ â€” **substantially done (2026-07-23):** `HotRegionList.SetHotRegion` now has `IGraphicsPath`-typed overloads (`SetHotRegion(object, params IGraphicsPath[])` / `SetHotRegion(object, PointF, params IGraphicsPath[])`), bridging to a fresh concrete `GraphicsPath` per path via `new GraphicsPath(path.PathPoints, path.PathTypes)` at the storage sink â€” same pattern as Chart's already-converted `HotRegionsList.AddHotRegion(IGraphicsPath, ...)` overloads. Confirmed (via `IGaugeDrawingResourceFactory.UnwrapPath`'s `Gdi` adapter implementing `IGraphicsPath.IsVisible`) that stored hot-region paths are consumed *only* by `CheckHotRegions`'s point-containment check, never redrawn, so the interface-typed storage copy doesn't need full rendering fidelity â€” just geometry. `CircularPointer`/`Knob`/`LinearPointer`'s private `hotRegions: GraphicsPath[2]` fields/`AddHotRegion(GraphicsPath, bool)` setters/`SetAllHotRegions` â€” the only real callers that already held an `IGraphicsPath` upstream (from B3's `NeedleStyleAttrib`/`BarStyleAttrib`/`MarkerStyleAttrib`/`KnobStyleAttrib`) and were unwrap-and-cloning it right back to concrete just to call this â€” retyped to `IGraphicsPath[]`/`AddHotRegion(IGraphicsPath, bool)`, removing the redundant unwrap/clone. One subtlety per class: where the source path's owning attrib object is `Dispose()`d in a `finally` block immediately after (`NeedleStyleAttrib`/`BarStyleAttrib`(Bar branch of `CircularPointer`)/`KnobStyleAttrib`), the interface-typed copy has to be taken *at that point* via `g.ResourceFactory.CreatePath(path.PathPoints, path.PathTypes)`, not deferred to `SetAllHotRegions` â€” otherwise the source is already disposed by the time it'd be read. Where no such disposal happens (`LinearPointer`'s Bar/Thermometer/Marker branches â€” `catch`-only, no `finally`), the `IGraphicsPath` reference is passed straight through and ownership transfers to `SetAllHotRegions`, which now disposes `hotRegions[0]`/`[1]` after handing them to `SetHotRegion` (previously just nulled the fields, since the concrete-`GraphicsPath` version had `HotRegionList` take ownership of the *same* array elements directly). `GaugeGraphics.DrawRadialSelection` (a draw, not hit-test-storage, call) and the ~10 remaining `SetHotRegion` call sites that build paths via still-concrete-only legacy methods (`GetFramePath`, `GetCircularRangePath`/`GetLinearRangePath`, etc. â€” see items 8/9) are unchanged; converting those belongs to converting those path-builders themselves, not this item. Verified: build 0 errors, `VisualRegressionTests` 79/79, `Chart.Rdl.Tests` 1/1, zero baseline diffs.

2. **`XamlRenderer.cs`/`XamlLayer.cs`** â€” architecturally blocked, not just undone. Needs three new primitives before any conversion is possible: a `ColorBlend`-equivalent gradient factory (arbitrary multi-stop gradients, no interface equivalent today), scale/shear-capable transforms (only rotate/translate are covered), and a way to build `IGraphicsPath`/`IBrush`/`IPen` without a live `GaugeGraphics` instance (its geometry-parsing methods run before any engine instance exists).

3. ~~**`CircularScale.DrawLabel`**~~ â€” **done (2026-07-23):** converted alongside its twin, `LinearScale.DrawLabel` (identical shape â€” same `Font`/`StringFormat`/`SolidBrush`/`Matrix`-rotation pattern, same fix). Both methods' `stringFormat`/`brush2`/shadow-`brush` locals are now `ITextFormat`/`IBrush` (via `graph.ResourceFactory.CreateTextFormat`/`CreateSolidBrush`/`GetShadowBrushResource`), reaching the existing `DrawString(string, IChartFont, IBrush, RectangleF, ITextFormat)` overload. `resizedFont` (from `GetResizedFont`, a `FontUnit`-aware resize needed before measurement) stays a concrete `Font` â€” it's still fed to the concrete `MeasureString(string, Font)` overload for layout math â€” and is bridged to `IChartFont` once via `graph.ResourceFactory.WrapFont(resizedFont)` right before the `DrawString` call sites (bridge-at-sink, not upstream). The rotation/shadow `Matrix` locals stay concrete, consistent with elsewhere in this doc (only round-trip through `graph.Transform`, never touch font/brush/format). Verified: build 0 errors, `VisualRegressionTests` 78/78 (zero baseline diffs â€” both sample gauges render visible tick-mark labels through this exact path, so this is a genuine pixel-verified conversion, not just "compiles"), `Chart.Rdl.Tests` 1/1.

4. ~~**`ScaleBase.GetLightBrush`'s `Circle` branch**~~ â€” **stale, resolved as a side effect of Chart's E1 work (2026-07-22):** `IPathGradientBrush.Blend`/`InterpolationColors` were added to the shared `Microsoft.Reporting.Rendering.IBrush.cs` (and implemented on both engines' `GdiPathGradientBrush`) while converting Chart's `DrawPieRel`, well before this item was re-checked. `GetLightBrush` (both the `Circle`/`PathGradientBrush` and linear-gradient branches) and its one caller, `ScaleBase.DrawTickMark`, are now fully converted: `GetLightBrush` retyped in place to return `IBrush`/take `IGraphicsPath` (only 1 real caller, so no dual-overload needed); `DrawTickMark`'s `graphicsPath`/`brush`/border `pen` locals are now `IGraphicsPath`/`IBrush`/`IPen`, feeding the already-interface-typed `FillPath(IBrush, IGraphicsPath, float, bool, bool)`/`DrawPath(IPen, IGraphicsPath)` overloads. `GaugeGraphics.DrawPathShadowAbs` got a new `IGraphicsPath`-typed sibling (only 1 real caller: this same method) since no interface-typed overload existed yet. The one non-obvious piece: the original code did a wholesale `((LinearGradientBrush)brush).Transform = matrix` assignment, which `ILinearGradientBrush` deliberately doesn't support (see `docs/decisions.md`'s "no generalized settable transform" decision) â€” resolved by observing that `matrix` in both real call sites (`CircularScale`/`LinearScale`'s `DrawTickMark` overrides) is *always* a composition of `Matrix.RotateAt(angle, absolutePoint)` calls around the same point, which is exactly reconstructible via `SetRotationTransform(angle, center)` once decomposed back into that (angle, center) pair via a new `ScaleBase.DecomposeRotation` helper (solves for the matrix's fixed point â€” exact, not an approximation, given the known single-rotation structure). Not exercised by any of the 3 existing sample-gauge visual-regression baselines (none enable tick-mark gradients), so 4 focused math tests were added (`ScaleBaseRotationTests.cs`) verifying the decomposition against known (angle, center) inputs and round-tripping through a fresh `RotateAt` to confirm bit-for-bit matrix reconstruction. Verified: build 0 errors, `VisualRegressionTests` 78/78 (74 + 4 new), `Chart.Rdl.Tests` 1/1, zero baseline diffs on the existing 55 gauge/chart baselines.

7. ~~**`DigitalSegment.cs`/`SegmentsCache.cs`**~~ â€” **done (2026-07-23):** converted in one atomic pass. Re-confirmed by solution-wide grep that `NumericIndicator.DrawSymbol` is the sole caller of every public entry point (`GetSegments`/`GetOrientedSegments`/`GetSymbol7`/`GetSymbol14`) â€” retype in place, no dual overload needed. Since `DigitalSegment` is `static` with no instance and hence no `ResourceFactory` field, every method in the call chain (down to the private `GetSegmentHKLN`/`GetSegment7`/`GetSegment14` helpers) now threads an `IGaugeDrawingResourceFactory` parameter from the caller's live `GaugeGraphics.ResourceFactory`, replacing every `new GraphicsPath()` with `resourceFactory.CreatePath()`. Rotation/shear matrices stay built via the native `Matrix` (`RotateAt`/`Shear` are exact, well-understood GDI+ operations not worth re-deriving) and are bridged to `Matrix3x2` only at the `IGraphicsPath.Transform(Matrix3x2)` call site itself via a `ToMatrix3x2` helper â€” the same literal 1:1-port convention already established by `Knob.cs`/`LinearPointer.cs`/`ScaleBase.cs`. `SegmentsCache` needed two adjustments since `IGraphicsPath` deliberately has no `Clone()` (not in Appendix A.1): its cached-copy semantics are reconstructed via `IGaugeDrawingResourceFactory.CreatePath(PointF[], byte[])`, feeding the cached path's own `PathPoints`/`PathTypes` back in; and its translate-only `matrix` field (GDI+'s `Matrix` reused via `Reset()`/`Translate()`) became a fresh `Matrix3x2.CreateTranslation(x, y)` built on each call, since `Matrix3x2` is a value type with no mutation methods (matching `GaugeGraphics.DrawPathShadowAbs`'s interface-typed sibling). At the one real call site, `NumericIndicator.DrawSymbol`, the digit `Brush` stays concrete (still fed by `GetFontBrush`, not the already-existing-but-unreachable `GetFontBrushResource`), so the returned `IGraphicsPath` is bridged back via `g.ResourceFactory.UnwrapPath(path)` right before `g.FillPath(Brush, GraphicsPath)` â€” bridge-at-sink, consistent with this doc's other conversions. No sample gauge previously exercised `NumericIndicatorStyle.Digital7Segment`/`Digital14Segment` (all 3 existing samples default to `Mechanical`), so a 4th sample (`SampleGauges.RenderDigital7SegmentNumericIndicator`, two indicators rendered together to exercise both a `SegmentsCache` cache-miss and cache-hit, plus the LED-dim "blank" segment path) and matching test (`Digital7SegmentNumericIndicator_MatchesBaseline`) were added; its baseline was generated by rendering the pre-conversion engine code via the git-stash technique (stash the 3 engine files, keep the new test files, render, copy the actual PNG into `Baselines/`, pop the stash) so it reflects pre-existing correct behavior, not the new code. Verified: build 0 errors, `VisualRegressionTests` 79/79 (78 + 1 new, zero baseline diffs â€” genuinely pixel-verified, not just build-verified), `Chart.Rdl.Tests` 1/1.
